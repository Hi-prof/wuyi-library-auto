# PRD：手动检查预约 & 一键签到所有账号

> 作用范围：`library-window` Windows 客户端的网页端控制台。
> 目标版本：与下一次发布同步。

## 1. 背景

- 客户端在后台维护两条定时链路：
  - 自动预约：`LocalAutomationPlanScheduler` 按每个计划的 `reserve_check_interval_minutes`（默认 30 分钟）轮询，发现“启用了自动预约但当天没有预约”的账号会自动补订。
  - 周期签到：`PeriodicAccountCheckinMonitor` 每 30 分钟为所有账号执行一次签到。
- 实际使用中出现两类痛点：
  1. 程序异常导致预约未成功，重启后当天没有预约。虽然 30 分钟内会自动补订，但用户希望可以**立即**手动触发一次补订，不必等到下一轮巡检。
  2. 调试或临时使用时，希望能一键对所有账号执行签到，而不是逐个进入“账号状态”页面点“立即签到”。

## 2. 目标

1. 在 **自动任务** 页面右上角增加“**手动检查预约**”按钮：
   - 立即对所有“启用了自动预约”的计划做一次预约状态巡检。
   - 已经有当天预约的账号跳过；缺失预约的账号自动补订（复用现有自动预约逻辑）。
   - 巡检结果在前端以 toast + 任务列表刷新的方式呈现。
2. 在 **账号状态** 页面右上角增加“**一键签到所有账号**”按钮：
   - 顺序对所有账号执行一次签到（复用现有 `/api/checkin` 流程）。
   - 在前端用 toast 汇总成功 / 失败数量，明细写入操作日志。
   - 与现有“检测当前状态”按钮并排，逻辑上互为补充：先一键签到，再检测状态查看结果。

## 3. 非目标

- 不修改自动任务和周期签到的现有定时频率与策略。
- 不引入新的数据持久化字段（除已有 `reserve_last_run_at` / `reserve_last_message` 等正常更新外）。
- 不调整账号配置、入口配置、座位查询等无关流程。
- 不实现“勾选部分账号批量签到”的高级筛选，按钮一次作用于全部账号。

## 4. 用户故事

- **U1**：作为一个开了自动预约的用户，当我发现今天没有预约时，可以打开“自动任务”页右上角点“手动检查预约”，让程序立刻检查并尝试补订，不用等到下一个 30 分钟节点。
- **U2**：作为多账号管理者，当我打开“账号状态”页面查看大家的预约 / 签到情况时，可以点右上角“一键签到所有账号”，让所有账号立刻执行一次签到，再点“检测当前状态”确认结果，省去逐个点“立即签到”。

## 5. 用户体验要求

### 5.1 自动任务页 · 手动检查预约

- 入口：自动任务页 `header.page-hero` 的 `hero-actions`，位于“新建自动任务”左侧。
- 按钮样式：`ghost-button`（次级），文本“手动检查预约”。
- 点击行为：
  - 按钮立刻进入忙态（`busy="true"`，禁用），文案变为“正在检查...”。
  - 后端返回后展示 toast：
    - 全部已有预约：`已检查 N 个计划，全部已有预约`。
    - 有补订成功：`已补订 X 个账号 / Y 个计划`，并把成功账号名拼在 toast 里（最多前 3 个，多余用 `等` 表示）。
    - 有失败：toast 用警告色，文本附带“X 个账号失败”，详情进入“调试中心”的操作日志。
  - 自动刷新自动任务卡片（`loadTasks(true)`）和账号状态卡片（`loadTaskStatuses(true)`），让用户看到 `最近一次自动预约` 字段被更新。
- 空数据兜底：
  - 没有任何启用了自动预约的计划时，按钮保持可点击，但点击后只弹 toast 提示“没有启用自动预约的计划”，不发请求或发请求后由后端返回该提示。
- 并发保护：
  - 当前已经有一次手动检查在进行时，按钮保持禁用状态，避免重复触发。
  - 按钮在“自动任务”视图未激活时保持可见，但隐藏在视图切换之外的请求由前端自行忽略。

#### 5.1.1 预约时段规则（重要差异）

- **当天**：无论计划里配置的 `start_hour` / `duration_hours` 是多少，手动检查统一按“**当前小时 → 学校允许的最晚结束时间**”下单：
  - `start_hour = 当前小时`（按小时向下取整，不做“过半向下一小时”的修正）。
  - `duration_hours = maxEndTime - start_hour`（`maxEndTime` 取学校返回的最晚结束时间，目前常见是 22）。
  - 例子：用户在 **15:40** 点“手动检查预约”，将预约 **15:00 ～ 22:00**（7 小时）。16:00 点击则是 16:00 ～ 22:00。
- **计划覆盖的未来日期（仅当 `continuous_reserve=true` 时有效）**：沿用现有 `build_automation_future_default_filters` 的逻辑，保持和自动定时巡检一致，避免行为漂移。
- **边界处理**：
  - 如果当前小时早于 `minBeginTime`（极早打开客户端的情况），`start_hour` 会上抬到 `minBeginTime`，再按上述规则算 `duration_hours`。
  - 如果当前小时 ≥ `maxEndTime - min_duration`（例如 22:30 之后），当天的 `duration_hours` < `min_duration`，该计划当天视为“无法补订”，归类为 `skipped`，message 写“当前时间已超过当天可预约窗口”。
  - 如果学校不允许跨整小时（与 `build_time_options` 返回的 `start_hour` 集合不符），会被 `validate_filters` 拦截，归类为 `failed`，message 透传校验错误。
- **不改计划本体**：手动检查只改这次请求的 `SearchFilters`，不会修改 `AutomationPlan.start_hour` / `duration_hours` 等持久化字段，保证下一次定时巡检仍按原配置运行。

### 5.2 账号状态页 · 一键签到所有账号

- 入口：账号状态页 `header.page-hero` 的 `hero-actions`，位于“检测当前状态”按钮**左侧**（即新增按钮在前、原“检测当前状态”按钮在后），保持原“检测当前状态”按钮的位置不变。
- 按钮样式：`ghost-button`，文本“一键签到所有账号”。
- 点击行为：
  - 没有账号时，按钮禁用并附 hint：`没有账号，先到账号管理页新建`。
  - 有账号时，按钮变忙态，文案“正在签到...”。
  - 后端返回后展示 toast：
    - 全部成功：`已为 N 个账号执行签到`。
    - 部分失败：`签到完成：成功 X，失败 Y`（警告色），失败详情写入操作日志。
  - 完成后刷新 `loadTaskStatuses(true)`，让账号状态卡片的“最近签到时间 / 预约状态”同步；同时调用 `loadAccounts(state.bootstrap?.selectedAccountName || "")` 让侧边栏“当前账号”和登录态徽标同步。
- 行为顺序：按 `state.accounts` 顺序逐个签到，避免并发对同一图书馆接口造成限流。
- 与每张账号状态卡片上现有的“立即签到”按钮共存，互不影响。
- 与“检测当前状态”按钮独立：一键签到只跑签到，不会自动触发状态检测，由用户决定是否再点“检测当前状态”查看结果。

> **注意**：账号管理页的 `hero-actions` 保持原样（仅“新建账号”按钮），不再额外增加按钮。

## 6. 边界与异常

| 场景 | 期望 |
| --- | --- |
| 没有任何计划/账号 | 按钮禁用或返回提示，不抛错。 |
| 某个计划缺少登录态 | 前端 toast 汇总时把该账号归类为“跳过”，操作日志记录 `未保存登录态`。 |
| 接口返回限流 | 复用 `wait_automation_reserve_gap("retry-after-rate-limit")` 的现有重试一次策略。 |
| 后端异常 | 返回 5xx 的标准错误结构，前端显示 toast，不影响其它功能。 |
| 同一账号同时被周期任务触发 | 现有 `recovery_lock` / `automation_scheduler` 内部锁负责互斥，不需要新增锁。 |
| 当天时间已过 `maxEndTime - min_duration`（例如 22:30 之后点击） | 当天 `duration_hours` 不足，归类为 `skipped`，toast 不标红；如 `continuous_reserve=true` 则继续尝试未来日期。 |
| 计划的 `selected_date` 为过去日期 | 复用现有 `build_automation_target_dates` 逻辑：该计划直接返回 `skipped`，message 为“当前入口没有可维护的预约日期”。 |

## 7. 验收标准

- [ ] 自动任务页右上角能看到“手动检查预约”按钮，且与“新建自动任务”按钮排列正确。
- [ ] 没有自动预约计划时，点击按钮显示明确提示，不会发起空请求。
- [ ] 至少 1 个计划缺失当天预约时，点击按钮可在 ~10 秒内完成补订并 toast 反馈。
- [ ] 在 15:40 点击“手动检查预约”，目标账号当天被预约为 15:00 ～ `maxEndTime`（通常 22:00），而计划本身的 `start_hour` / `duration_hours` 不被改写。
- [ ] 账号状态页右上角能看到“一键签到所有账号”按钮，位于“检测当前状态”左侧；没有账号时按钮禁用。
- [ ] 账号管理页 `hero-actions` 保持原样（仅“新建账号”按钮），没有新增按钮。
- [ ] 一键签到成功后，账号状态页对应账号的“最近签到时间/消息”刷新，与“立即签到”单账号触发结果一致。
- [ ] 现有自动定时巡检和周期签到的行为保持不变（运行时间间隔、计划存储格式、字段含义无回归）。

## 8. 度量与可观测性

- 操作日志：两个新接口的请求 / 响应都通过现有 `requestJson` / `requestJsonWithBody` 的 `debugLabel` 写入“调试中心”。
- 文案明确：toast 中带统计数据，便于排障时直接复述给开发。
- 统计字段：响应体里返回 `checkedCount` / `bookedCount` / `skippedCount` / `failedCount` 等，方便后期接入更多 UI。

## 9. 上线影响

- 新增 2 个 HTTP 接口、2 个 UI 按钮、若干文案。
- 不影响现有数据迁移、配置文件、登录态文件。
- 仅 `library-window` 子项目，受影响测试集中在 `library-window/tests/`。
